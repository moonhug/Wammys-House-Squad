## 1. 기반 지식: Mono vs il2cpp

본격적인 분석에 앞서, 우리가 어떤 환경과 직면하고 있는지 그 구조를 먼저 명확히 이해할 필요가 있습니다. Unity 게임의 동작 방식은 크게 Mono와 il2cpp 두 가지로 나눌 수 있습니다.

우선 **Mono 빌드**는 아주 고전적이고 우리에게 친숙한 구조를 가집니다. 작성된 C# 코드가 중간 언어인 IL로 컴파일되며, 게임이 실행될 때 JIT(Just-In-Time) 컴파일러를 통해 기계어로 번역되어 동작하게 됩니다. 이 방식의 가장 큰 약점이자 분석가 입장에서의 장점은, 빌드 결과물에 `Assembly-CSharp.dll`이라는 원본의 뼈대가 고스란히 남는다는 점입니다. 그렇기에 DnSpy 같은 툴에 파일을 올리기만 하면 내부 코드가 훤히 들여다보일 만큼 분석이 매우 쉬워 사실상 보안에는 취약하다고 볼 수 있습니다.

![Mono 기반 스크립트 런타임 동작 흐름 다이어그램](./pdf/003.png)

하지만 **il2cpp 빌드**는 완전히 다른 철학을 보여줍니다. 이름에서 알 수 있듯, C# 코드를 IL 코드로 변환한 뒤 이를 다시 **C++ 코드**로 역변환시켜 최종 네이티브 기계어로 빌드해 버립니다. (AOT, Ahead-Of-Time 방식)

![il2cpp 기반(C++ 변환) 동작 흐름 다이어그램](./pdf/004.png)

이 무지막지한 과정으로 인해 빌드된 게임의 `il2cpp_data` 폴더를 아무리 뒤져보아도 `Assembly-CSharp.dll` 파일은 흔적조차 찾을 수 없습니다. 모든 코드는 C++ 기반의 네이티브 어셈블리 형태로 굳어져 있기에 일반적인 방법으로는 구조 자체를 파악할 수단이 차단되어 보안 방어력이 매우 뛰어납니다.

![실제 il2cpp_data 폴더 내부 튜토리얼 캡쳐 - Assembly-CSharp.dll이 없는 화면](./pdf/005.png)

## 2. 잃어버린 DLL을 찾아라: 분석 도구 콤보 (Tool Tool Tool..)

`Assembly-CSharp.dll`이 소실된 il2cpp 빌드를 처음 마주하게 되면 눈앞이 캄캄하겠지만, 다행히도 강력한 툴체인을 하나로 엮어서 이 난관을 헤쳐나갈 수 있습니다.

첫 번째로 구세주 역할을 해주는 툴은 **il2cppinspector**입니다. 이 툴은 게임 내부에 남아있는 Json 형태의 메타데이터를 정밀하게 분석하여, 함수와 클래스 껍데기만 존재하는 빈 `.NET assembly shim DLL`을 원본과 유사하게 복원해 냅니다. 이뿐만 아니라 복잡한 디스어셈블러 환경에서 알아보기 힘든 함수명을 매칭해 줄 수 있는 Python rename 스크립트까지 함께 제공해 주는, 일련의 과정에서 가장 핵심적인 초석을 다지는 단계입니다.

![il2cppinspector 실행 화면 및 메타데이터 추출 세팅 UI 캡쳐](./pdf/006.png)

이렇게 복원된 껍데기(Dummy DLL)는 곧바로 **DnSpy**의 훌륭한 분석 재료가 됩니다. 껍데기뿐인 허울이기에 실제 구현된 C# 코드는 당연히 비어 있지만, 게임 전체를 구성하는 클래스 구조와 함수 형태가 적나라하게 드러나게 됩니다.

![DnSpy에 Dummy DLL을 올렸지만 내부 코드를 볼 수 없음을 보여주는 화면](./pdf/007.png)

여기서 우리가 취해야 할 가장 중요한 수확은, 함수의 원본 로직은 확인할 수 없더라도 해당 함수가 네이티브 메모리 상의 어느 지점에 위치하고 있는지를 가리키는 **RVA(Relative Virtual Address)와 VA Offset 값**이라는 힌트를 확보할 수 있다는 것입니다.

![DnSpy에서 해당 함수의 RVA, Offset 주소 값이 표시된 영역을 강조한 이미지](./pdf/008.png)

마지막으로 DnSpy에서 얻은 구체적인 Offset 주소라는 보물지도를 들고 어셈블리의 심연인 **IDA Pro**에 진입하게 됩니다. 게임의 실제 코드 로직이 오롯이 담긴 네이티브 엔진 파일인 `GameAssembly.dll`을 IDA에 올린 뒤, 획득한 주소로 곧장 이동하여 해당 로직의 실제 어셈블리어를 추적하고 디컴파일합니다.

![IDA Pro에서 수많은 분석 불가능 함수 목록이 나열된 모습의 캡쳐](./pdf/009.png)

여기에 더해, `IDA-pro-mcp`와 Gemini CLI를 등에 엎고 `cash`, `gold`, `money`와 같은 재화 관련 핵심 키워드(예: `AddCash` 등)를 방대한 코드 내부에서 집중적으로 탐색하면 분석 속도와 효율을 획기적으로 끌어올릴 수 있습니다.

![터미널에서 IDA-pro-mcp 및 Gemini CLI로 'cash' 관련 함수 키워드를 검색한 결과 화면](./pdf/012.png)

## 3. 실제 게임 분석: Bloons TD6 사례 (Game Analyze)

앞선 복구와 매핑 과정(il2cppinspector -> DnSpy -> IDA)을 성공적으로 마쳤다면, 이제 게임 내에서 실제 수치 데이터가 움직이는 생동감 있는 로직을 정밀 타격할 차례입니다. 디펜스 게임인 Bloons TD6를 분석한 사례를 살펴보겠습니다.

![Bloons TD6 인게임 플레이 장면 캡쳐](./pdf/014.png)

가장 먼저 해야 할 일은 인게임 내에서 직관적으로 확인할 수 있는 명확한 **수치 변동**의 흐름을 포착하는 것입니다. 예를 들어 보유 캐시가 372원인 상태에서 특정 타워를 판매해 119원의 캐시를 얻어 최종 수치가 491원으로 변하는 그 찰나의 연산 구간을 타겟으로 잡습니다.

![인게임 상 타워 판매에 따른 캐시 변동 (372 -> 491) 및 119원 증가 부분 강조 이미지](./pdf/015.png)

인위적으로 자금이 더해지는 액션 직후, 앞서 IDA로 매핑해둔 타겟 메모리 주소(예: `GameAssembly.dll+4C7F3D`) 부근의 어셈블리 흐름에 집중합니다. 이 과정에서 특정 동작이 실행될 때 어떤 레지스터에 값이 누적되고 어떻게 차감이 일어나는지 추적합니다. 특히 `AddCash`라는 실질적인 인터페이스가 어떻게 호출되어 메모리의 값을 덮어쓰는지 상세히 파악하게 되면, 굳건히 닫혀있던 게임 내부 경제 로직의 민낯을 온전히 들여다볼 수 있게 됩니다.

![IDA의 GameAssembly.dll 디버깅 뷰에서 Offset 주소와 레지스터 변화를 추적하는 화면](./pdf/020.png)

## 마무리

결과적으로 il2cpp 빌드 시스템은 기존 Mono 환경과 다르게 런타임 코드와 심볼 단서가 네이티브에 깊숙히 은닉되어 있기 때문에 분석가 입장에선 아주 까다로운 허들입니다.

하지만 아무리 두꺼운 벽이라 할지라도, **il2cppinspector**를 통해 설계도(Metadata)를 스케치해내고, **DnSpy**를 통해 전체적인 구조와 좌표(Offset)를 식별한 뒤, 마침내 **IDA**를 통해 실질적이고 구체적인 네이티브 코드(`GameAssembly.dll`)를 역어셈블한다는 3단 콤보 방법론을 활용하면 충분히 파훼가 가능합니다. 원본 코드를 상실한 il2cpp 게임이라도 그 깊은 코어 로직(재화 획득 등)까지 이 파이프라인을 그대로 적용하여 성공적으로 추적 및 분석할 수 있음을 확인했으며, 이는 앞으로의 다양한 Unity 게임 분석에서도 언제나 강력한 길잡이가 되어 줄 것입니다.
